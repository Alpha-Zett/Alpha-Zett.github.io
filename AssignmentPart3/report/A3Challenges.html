Plans change

Several changes were made to our initial plan:

Features

We started with five minimum viable features: Interactive Map, Case presentation, Chatbot, COVID-19 Trend and RSS Feed. As our project progressed, our team decided to implement some changes to deliver more value to the end-users. As a result, minimum viable feature two is now ‘Contact Tracing’ and we came up with three extended features: Regional RSS Feeds, COVID Cases Presentation and Testing Facility Locations.

Testing

The team initially planned to create a prototype website and have it tested with 12 ~ 15 unique users. We did not have much time to conduct proper testing because some features were more difficult to develop and therefore took longer than expected. To overcome this, our group had all our features tested individually with different users.

Timetable realistic?

Our initial timetable was realistic for the most part. The timetable was constructed based on a logical progression and allocated reasonable times for the planning, research and designing phases. However, there are several changes that we would make knowing what we know now.

Development and Testing

The timetable we created for the development and testing phases were vague, which resulted in the team spending more time on the development phase and conducting a small number of tests. We would improve this by defining deadlines for micro-goals leading up to the full development of the features. Doing so can help the team work more efficiently, and allow time for the team to cross-check each other’s features to see if they are meeting expectations and provide feedback for improvement. It will also allow more time for testing, which gives the team the opportunity to further improve on the products, and learn more things.

Assignments

The timetable for assignments was also vague, which resulted in the team making inconsistent progress throughout the weeks and leaving tasks to the last week. We think that this can be improved by breaking down assignments to smaller tasks and aim to complete some tasks each week. This can help improve our work efficiency as our members are not overwhelmed by the amount of work that needs to be accomplished.

Tools and Technologies

The Tools and Technologies we used worked out for the most part. However, there were some that did not work out as expected:

Adobe XD

Matthew initially used Adobe XD as he had access to it and wanted to see if it were easier to work with using native Adobe files, and wanted to test out what the preview of the prototype was like and more importantly how the final preview is generated and to see how usable this is in the sense of being able to add iframe code into a HTML page to incorporate into an assignment submission. This exercise is comparing this software with the software that I started to learn in the previous semester - Figma.

It was an interesting exercise to carry out as his initial assumptions were that Adobe XD would be easier to use, more precise and would have realistic animations and transitions to recreate convincing user experiences from the prototype environment.

What I found is that it was more clunky to use, even though it had very similar workflows, Figma was easier to use, it felt smoother to build up the pages, handled all types of files similarly and had superior animations or transitions that could be carried out quickly.
