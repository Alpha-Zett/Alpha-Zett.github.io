Collaborative Workspaces

Figma
Figma is an online website used for editing graphics and creating prototypes. Our group decided to use Figma as it is very comprehensive and easy to use and allows us to easily create prototypes for our website. We selected Figma among the alternatives as our group is familiar with using the software. Alternatives include: Invision, Axure RP and Lucid Chart

GitHub
GitHub is an online software used for hosting and software development. Our group decided to use this because it allows us to easily share HTML and CSS code and preview the code on a browser.

Visual Studio Code
Visual Studio Code is a source-code editor which provides support debugging, syntax highlighting and automatic code completion. Though source-code can also be edited on GitHub, we chose to also use this software because it allows us to quickly identify and fix mistakes written on our code. An alternative is: Brackets.io

Group Contract

Ground Rules

  • Attend as many meetings as possible. If you cannot make it, please message the group at least 24 hours beforehand.
  • Our group will host 1~2 meetings per week.
    - Monday afternoon – Reflect on tasks completed the previous week and planning what is to come ahead
    - Friday mornings – Update each other on progress made throughout the week and any questions or concerns that need to be clarified.
  • Our group will communicate each other at least once a day via Microsoft Teams. We will update each other on micro-tasks accomplished, and/or ask questions or concerns about their tasks. We will also use this platform to help each other in case someone needs help doing their allocated tasks.
  • Everyone in the group will respect each other.
  • Everyone will always their best.

Dealing with setbacks and/or unmet expectations

  • We will try to deal with the setback and/or unmet expectations by communicating with the team through Microsoft Teams
  • If the group cannot solve it ourselves, we will ask our mentor for guidance and suggestions for ways to solve the issue.
  • Our team will regularly communicate their progress. Doing so will allow everyone to identify any problems that may affect progress or the outcome of the project.

Providing Feedback

  • We will do our best to have at least 2 to 4 members of the group read through our parts, before finalizing it.
  • When giving feedback, we will provide critical feedback and suggest ways in which they can improve their parts.
  • The tasks do not have to be “almost complete” before asking for feedback. Everyone can ask feedback about anything at any reasonable time during the project. (e.g. Asking for feedback about an approach before commencing.)
  • Make sure ask for feedback at least 24-48 hours before the agreed deadline.
